# 2026-03-31

1. **Pay your debt the day you take it.** You left three separate comments saying "this should be refactored" and moved on. That habit will rot any codebase. If you know it's wrong, fix it before committing.

2. **A class that does 6 things is 5 classes you haven't written yet.** `arena.py` and `troop.py` are doing everyone else's job. Split them.

3. **Never use `assert` for runtime logic.** Assertions are for catching your own mistakes in dev — they're not error handling. If pathfinding fails in a shipped game, you want recovery, not a crash.

4. **Magic numbers are just undocumented decisions.** Every hardcoded `16` or `30` is a future bug waiting to happen when you change one and forget the other eleven.

5. **Delete dead code, don't comment it out.** `if False:` blocks and commented-out sprite loading aren't version control — that's what git is for. Dead code is noise that erodes trust in the codebase.

6. **Don't copy-paste and change one character.** The troop render methods are the textbook definition of when to abstract. If you're changing a single letter across three identical functions, you've already failed DRY.

7. **The TODO list in your code is a lie you tell yourself.** You have 5 TODOs, some with "remove the hardcoding" that never got removed. Either fix it now or accept it's permanent — but don't pretend the comment makes it okay.

The underlying issue isn't skill — the code works and the structure shows real understanding. It's **follow-through**. You spot the problems, documents them, and ships anyway. That's the one habit worth fixing above everything else.
